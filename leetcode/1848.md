# LeetCode 1848. Minimum Distance to the Target Element

題目係[呢度](https://leetcode.com/problems/minimum-distance-to-the-target-element/)

俾你一個 array `nums`、一個 `target` value 同一個 `start` index，搵一個 index `i` 令到 `nums[i] == target` 而 `abs(i - start)` 最細。

## 解法

用 zig-zag scan：由 `start` 出發，交替向右同向左展開，第一個搵到 `target` 嘅就係答案。

Zig-zag 嘅 offset sequence 係 `0, 1, -1, 2, -2, 3, -3, ...`，呢個 sequence 可以用一個 recurrence 表達（點樣 construct 呢類 recurrence 可以參考[呢篇](https://github.com/ceciliachan1979/math/blob/main/Misc/Obfuscate/Obfuscate.tex)）：

$$s_n = -s_{n-1} + s_{n-2} + s_{n-3}$$

initial values 係 $s_0 = 0, s_1 = 1, s_2 = -1$。

用 recurrence 嘅 motivation 係想 avoid loop body 入面嘅 control structure — 唔使 if-else 去決定方向同步長，一個 formula 搞掂晒。

將個 recurrence rewind 三步，initial values 變成 $(-1, 1, 0)$，咁就可以用 recurrence 直接 generate 成個 sequence，連 initial conditions 嘅 special case 都唔使 check。段 code 完全冇 control structure（除咗個 while loop 同 bounds check）：

```py
class Solution:
    def getMinDistance(self, nums, target, start):
        n = len(nums)
        a, b, c = -1, 1, 0
        while True:
            a, b, c = b, c, -c + b + a
            i = start + c
            if 0 <= i < n and nums[i] == target:
                return abs(c)
```

老實講，我覺得呢段 code 係 optimal，但喺 LeetCode benchmark 仲係慢咗 1ms。我覺得只係 noise — test cases 細，measurement 唔穩定。Cache miss 都唔似，因為 access 全部都係 local 嘅。

## Output Sensitive Analysis

呢個 solution 嘅 time complexity 唔係 $O(n)$，而係 $O(\text{ans})$ — 取決於答案本身嘅大小。如果 target 就喺 `start` 隔離，根本唔使 scan 成個 array。

呢個就係所謂嘅 output sensitive algorithm：complexity 取決於 output 而唔係 input size。

## Lower Bound Proof

$O(\text{ans})$ 係 optimal，可以用 adversary argument 證明。

如果一個 algorithm 冇 evaluate 所有距離細過 `ans` 嘅位置，adversary 可以將 `target` 放喺其中一個未 evaluate 嘅位置，令到 algorithm 嘅答案係錯嘅。所以任何 correct algorithm 都必須 evaluate 至少 $\Omega(\text{ans})$ 個位置，呢個就係 lower bound。
